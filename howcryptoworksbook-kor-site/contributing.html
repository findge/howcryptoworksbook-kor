기여 가이드

이 책은 여기 GitHub에서 공동으로 공개적으로 개발되었습니다. 우리는 모든 사람의 의견, 기여 및 수정 사항을 받아들입니다.

프로젝트 현황

첫 번째 버전(리뷰어 모집)

우리는 공식 초판이 출시되기 전에 콘텐츠를 다듬는 데 도움을 줄 전문 리뷰어와 기여자를 적극적으로 찾고 있습니다.

기여하는 이유

이는 암호화폐 시장, 인프라 및 위험을 이해하기 위한 확실한 고급 리소스를 만들기 위한 오픈 소스 노력입니다. 기여함으로써 귀하는 업계가 차세대 참가자와 개발자를 교육하는 방법에 대한 기준을 높이는 데 도움이 됩니다.

기술적인 수정이든, 새로운 예이든, 설명의 의미 있는 개선이든 귀하의 기여가 승인되면 당신은 책의 출판판에서 리뷰어로 기록될 것입니다. 이는 최종 텍스트 작성에 있어 귀하의 전문 지식과 역할을 영구적으로 인정하는 것입니다.

우리는 이 업계를 발전시키는 가장 좋은 방법은 개방적이고 엄격하며 협력적인 지식 공유를 통해서라고 믿으며, 모든 의미 있는 기여는 이를 실현하는 데 도움이 됩니다.

라이선스 및 귀속

모든 기여에는 적절한 라이선스가 부여되고 귀속되어야 합니다. 자신의 독창적인 저작물을 제공하는 경우 CC-BY 라이선스(크리에이티브 커먼즈 저작자 표시)에 따라 제공하는 것입니다. 귀하는 기여에 대한 귀속으로 감사의 글에 자신의 이름이나 가명을 추가할 책임이 있습니다.

다른 곳에서 기여를 소싱하는 경우 호환되는 라이선스가 있어야 합니다. 책은 처음에 CC-BY-NC-ND 라이선스에 따라 출시됩니다. 즉, 기여는 MIT, CC0, CC-BY 등과 같은 공개 라이선스에 따라 라이선스가 부여되어야 합니다. 다음과 같이 기여 위에 마크다운 설명을 포함하여 원본 출처와 원본 라이선스를 표시해야 합니다.

Pull Request로 기여하기

이 책에 기여하는 가장 좋은 방법은 끌어오기 요청을 하는 것입니다.

  1. GitHub 계정으로 로그인하거나 지금 계정을 만드세요.
  2. 이 저장소를 포크합니다. 포크로 작업하세요.
  3. 변경하려는 새 분기를 생성합니다(예: git checkout -b my_contribution). 또는 develop 분기에서 변경을 수행합니다.
  4. 파일당 하나의 PR - 대규모 병합을 방지하려면 마크다운 파일당 하나의 풀 요청을 수행하세요.
  5. 변경하려는 부분의 마크다운 파일을 편집하세요.
  6. 귀속 - 기여에 대한 귀속을 원할 경우 풀 요청 설명에 GitHub 사용자 이름이나 가명을 적어주세요. 향후 버전에서는 중요한 기여자가 인정될 것입니다.
  7. 변경 사항 커밋 - 수정 내용을 설명하는 커밋 메시지를 포함합니다.
  8. 이 저장소에 대해 풀 요청을 제출하세요.

기여할 내용

우리는 특히 다음을 환영합니다.

  • 기술적 수정 - 프로토콜, 메커니즘 또는 개념이 설명되는 방식의 오류
  • 오래된 정보 - 암호화폐는 빠르게 움직입니다. 콘텐츠를 최신 상태로 유지하는 데 도움을 주세요.
  • 명확성 개선 - 복잡한 주제에 대한 더 나은 설명
  • 컨텍스트 누락 - 우리가 간과했을 수 있는 중요한 주의 사항이나 고려 사항
  • 코드 예제 - 코드 조각 또는 기술 예제 수정

제출하지 말아야 할 것

다음과 같은 경우에는 PR을 제출하지 마세요.

  • 사소한 오타 또는 문법 문제(이러한 사항은 사본 편집 중에 처리됩니다)
  • 명확성을 향상시키지 않는 스타일 선호
  • 프로모션 콘텐츠 또는 링크 추가

문제에 기여하기

실수를 발견했는데 수정 방법을 모르거나 끌어오기 요청을 수행하는 방법을 모르는 경우 문제를 제출할 수 있습니다. 문제를 제출하면 문제를 확인하고 해결하는 데 도움이 됩니다.

문제를 제출할 때 다음을 포함하십시오.

  • 문제가 나타나는 장과 섹션
  • 문제에 대한 명확한 설명
  • 해당되는 경우 올바른 정보가 되어야 한다고 생각하는 내용
  • 수정을 지원하는 모든 소스

기술 지침

줄 끝

모든 제출물은 Unix와 같은 줄 끝인 LF(CR/LF 아님)를 사용해야 합니다. 잘못된 줄 끝 또는 줄 끝의 변경은 diff 도구에 혼란을 야기하고 전체 파일이 변경된 것처럼 보이게 만듭니다.

확실하지 않거나 OS로 인해 작업이 어려운 경우 VSCode와 같은 개발자의 텍스트 편집기를 사용하는 것이 좋습니다.

마크다운 형식

  • 표준 GitHub 스타일의 Markdown 사용
  • 일관된 제목 계층 구조 유지(장 제목은 #, 섹션은 ##, 하위 섹션은 ###)
  • 처음 소개할 때 핵심 용어에 굵게를 사용하세요.
  • 기술 용어, 주소, 함수 이름 등에 대해서는 code formatting을 사용합니다.
  • diff에서 가독성을 높이기 위해 줄을 적당한 길이로 유지하십시오.

커밋 메시지

명확하고 설명이 포함된 커밋 메시지를 작성하세요.

Good: "Fix incorrect explanation of ECDSA nonce generation in ch05_custody"
Bad: "Fixed stuff"

검토 과정

  1. 모든 풀 요청은 관리자가 검토합니다.
  2. 기술 기여는 확인을 위해 도메인 전문가에게 전송될 수 있습니다.
  3. 병합하기 전에 변경이나 설명을 요청할 수 있습니다.
  4. 책에서 상당한 공헌을 인정받을 것입니다.

행동 강령

  • 모든 상호작용에서 존중하고 건설적이어야 합니다.
  • 기여자가 아닌 콘텐츠에 집중하세요
  • 기여와 피드백에 대해 선의를 가지고 있다고 가정합니다.
  • 주제에 관한 전문적인 토론을 유지하세요.
